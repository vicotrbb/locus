# Experiment 0226: Remote-Free Service Telemetry Stability Manifest

Date: 2026-07-03

## Postulate

Postulate:
`documentation/postulates/0218-remote-free-service-telemetry-stability-manifest.md`

The postulate said that a small line-oriented manifest could make repeated
remote-free service telemetry stability checks reproducible without weakening
the existing counter gate.

## Change

Added manifest parsing to `remote_free_service_timing_stability.rs` in
`locus-validate`.

The manifest format is plain text:

```text
# role label path
baseline apply-confirm-a apply-confirm-a.txt
candidate apply-confirm-b apply-confirm-b.txt
candidate apply-confirm-drift apply-confirm-drift.txt
```

Blank lines and `#` comments are ignored. Each data row has exactly three
whitespace-separated fields: role, label, and path. The role must be
`baseline` or `candidate`. The parser rejects empty manifests, unknown roles,
duplicate labels, duplicate baselines, missing baselines, missing candidates,
and malformed rows.

The `remote_free_service_sample_compare` example now accepts:

```text
--manifest <manifest>
```

Manifest entry paths are resolved relative to the manifest path when they are
not absolute.

## Commands

```text
cargo test -p locus-validate remote_free_service -- --nocapture
printf '%s\n' '# role label path' 'baseline apply-confirm-a apply-confirm-a.txt' 'candidate apply-confirm-b apply-confirm-b.txt' 'candidate apply-confirm-drift apply-confirm-drift.txt' > target/locus-evidence/remote-free-service-sample-compare/apply-confirm-manifest.txt
cargo run -p locus-validate --example remote_free_service_sample_compare -- --manifest target/locus-evidence/remote-free-service-sample-compare/apply-confirm-manifest.txt
cargo run -p locus-validate --example remote_free_service_sample_compare -- target/locus-evidence/remote-free-service-sample-compare/apply-confirm-a.txt target/locus-evidence/remote-free-service-sample-compare/apply-confirm-b.txt target/locus-evidence/remote-free-service-sample-compare/apply-confirm-drift.txt
```

## Results

The manifest command over the real saved apply-confirm outputs reported:

```text
remote_free_service_telemetry_timing_stability=mixed baseline=apply-confirm-a candidate_runs=2 accepted_runs=1 discarded_runs=1 timing_ranges=1
remote_free_service_telemetry_timing_discard run=apply-confirm-drift drift_entries=1
remote_free_service_telemetry_timing_range benchmark=remote_free_service_runtime_apply_confirm range_runs=2 min_estimate_ps=56595000 max_estimate_ps=56867000 spread_ps=272000
```

The positional command over the same files reported the same stability status,
candidate count, accepted count, discarded count, timing range count, and
timing range:

```text
remote_free_service_telemetry_timing_stability=mixed baseline=target/locus-evidence/remote-free-service-sample-compare/apply-confirm-a.txt candidate_runs=2 accepted_runs=1 discarded_runs=1 timing_ranges=1
remote_free_service_telemetry_timing_discard run=target/locus-evidence/remote-free-service-sample-compare/apply-confirm-drift.txt drift_entries=1
remote_free_service_telemetry_timing_range benchmark=remote_free_service_runtime_apply_confirm range_runs=2 min_estimate_ps=56595000 max_estimate_ps=56867000 spread_ps=272000
```

Focused unit tests covered manifest parsing with comments and blank lines,
duplicate labels, duplicate baselines, unknown roles, missing baselines,
missing candidates, and malformed rows. The focused remote-free service
validator test count increased from 17 to 24.

## Interpretation

The postulate survived.

Manifest mode preserved the exact timing evidence from the positional command
while replacing long file-path labels with stable manifest labels. The
controlled drift run was still discarded and did not contribute timing
evidence.
